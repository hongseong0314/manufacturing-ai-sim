# MES Domain Model

Status: canonical  
Last updated: 2026-05-06

## Purpose

This document defines the MES domain vocabulary and how it maps to the current
simulator. It is the data model reference for APIs, stores, UI screens, and
future normalized database work.

## Simulator To MES Mapping

| Simulator concept | MES concept | Current field/source |
|---|---|---|
| `Task` | Wafer-level work item | `src/objects.py` |
| `Task.job_id` | Lot id / work order id | `job_id` |
| `Task.uid` | Wafer id source | `WAFER_{uid}` |
| `Task.due_date` | Commit / due date | `due_date` |
| `Task.spec_a`, `Task.spec_b` | Operation spec/control limit | process specs |
| `Task.location` | Current operation/location | `QUEUE_A`, `PROC_*`, `COMPLETED` |
| `Task.realized_qa_A/B` | Metrology / QA result | process event logs |
| `Task.rework_count` | Rework count | task state |
| `Task.customer_id` | Customer id | generated by `DataGenerator` |
| `Task.material_type`, `Task.color` | Product/packing attributes | generated by `DataGenerator` |
| `Task.margin_value` | Business priority signal | generated by `DataGenerator` |
| `Process*_Machine` | Equipment | machine snapshots |
| `u`, `m_age` | A consumable and machine age | A equipment health |
| `v`, `b_age` | B solution and bath age | B equipment health |
| `get_decision_state()` | Decision snapshot | AI/rule input |
| `env.step(actions)` | Command execution adapter target | simulator action application |

## Current Dataclasses

The MES DTOs live in `src/mes/domain.py`.

### Product

Product is the business and manufacturing classification of work.

Current fields:

- `product_id`
- `product_family`
- `priority_class`
- `default_route_id`
- `spec_profile_id`
- `customer_id`
- `margin_value`

Current simulator source is partial. `Task.material_type`, `Task.color`,
`Task.customer_id`, `Task.margin_value`, and specs should become richer Product
or ProductProfile data over time.

### Lot

Lot is the MES grouping around a set of wafers/tasks.

Current fields:

- `lot_id`
- `product_id`
- `route_id`
- `current_operation_id`
- `carrier_id`
- `status`
- `priority`
- `due_date`
- `quantity`
- `rework_count`

Current source: grouped `Task.job_id` rows in `SimulatorMESAdapter`.

### Wafer

Wafer represents one simulator task.

Current fields:

- `wafer_id`
- `lot_id`
- `slot_no`
- `status`
- `current_operation_id`
- `qa_results`
- `genealogy_parent_ids`
- `genealogy_child_ids`
- `task_uid`

Current id mapping:

```python
wafer_id = f"WAFER_{task.uid}"
```

### Equipment

Equipment maps to A/B/C machines.

Current fields:

- `equipment_id`
- `equipment_group_id`
- `status`
- `current_lot_id`
- `current_recipe_id`
- `capable_operations`
- `batch_size`
- `health_state`
- `last_event_time`

Status mapping:

| Simulator | MES |
|---|---|
| `idle` | `IDLE` |
| `busy` | `RUN` |

Health state includes A `u/m_age`, B `v/b_age`, C `finish_time`, and other
machine snapshot fields.

### Recipe

Recipe is the process-control master and runtime selection record.

Current default recipes:

| Recipe id | Operation | Parameters |
|---|---|---|
| `SIM_A_BASE` | A | temp, flow, duration |
| `SIM_B_DEFAULT` | B | chem_a, chem_b, time |
| `SIM_C_NO_RECIPE` | C | empty parameter set |

C has no physical recipe today, but the domain keeps a C recipe record so API and
UI can treat all stages consistently.

### FeatureSnapshot

FeatureSnapshot captures model/rule input.

Fields:

- `feature_snapshot_id`
- `correlation_id`
- `layer_id`
- `source`
- `decision_state`
- `features`

Each layer must create a snapshot with layer-specific features.

### AIRecommendation

AIRecommendation is the common envelope for all layer outputs.

Fields:

- `recommendation_id`
- `recommendation_type`
- `layer_id`
- `objective_id`
- `policy_id`
- `model_id`
- `model_version`
- `feature_snapshot_id`
- `correlation_id`
- `candidate_actions`
- `recommended_action`
- `score`
- `confidence`
- `reasons`
- `parent_recommendation_id`
- `rule_validation_status`
- `rule_validation_reasons`
- `final_command_id`

The envelope is intentionally generic. The layer-specific contract lives in
`candidate_actions` and `recommended_action`.

### Event

Event is the audit timeline.

Important event types today:

- `OBJECTIVE_SELECTED`
- `STAGE_PRIORITY_UPDATED`
- `DISPATCH_RECOMMENDED`
- `PACK_RECOMMENDED`
- `RECIPE_RECOMMENDED`
- `RULE_VALIDATION_PASSED`
- `RULE_VALIDATION_REJECTED`
- `COMMAND_CREATED`
- `COMMAND_EXECUTED`

The process environments also have simulator event logs such as
`task_assigned`, `task_completed`, `task_queued`, `pack_started`, and
`pack_completed`.

### MESCommand

MESCommand is the executable command envelope.

Current command type:

- `RESERVE_AND_TRACK_IN`

Current fields:

- `command_id`
- `command_type`
- `correlation_id`
- `validation_status`
- `status`
- `validated_command`
- `simulator_actions`
- `reasons`

The command may contain a simulator action payload after validation.

### Genealogy

Genealogy is defined but not fully populated yet. It should connect:

```text
Product -> Lot -> Wafer -> Operation -> Equipment -> Recipe -> QA result
  -> Recommendation -> Validation -> Command -> Event
```

## Persistence

### Current Store

`InMemoryMESStore` currently stores:

- lots,
- wafers,
- equipment,
- recipes,
- feature snapshots,
- recommendations,
- validations,
- commands,
- events.

`SQLiteMESStore` persists these records as JSON payloads in local SQLite tables.
This is a practical MVP store, not the final normalized database.

### Target Store

The production-style repository should normalize at least:

- products,
- lots,
- wafers,
- equipment,
- recipes,
- feature snapshots,
- recommendations,
- rule validations,
- commands,
- events,
- genealogy,
- reservations/locks,
- operator approvals.

Until event replay is implemented, runtime state can be synced from simulator
snapshots into the store. The target MES should eventually rebuild WIP and
genealogy from events.

## Data Ownership

| Data | Owner |
|---|---|
| physics state | `src/environment/*` |
| simulator task/machine snapshots | `ManufacturingEnv.get_decision_state()` |
| MES DTO conversion | `SimulatorMESAdapter` |
| runtime entity persistence | `InMemoryMESStore` / `SQLiteMESStore` |
| audit persistence | store classes |
| final command validation | `MESRuleEngine` |
| UI presentation | API response contracts and `live_ui.py` |
