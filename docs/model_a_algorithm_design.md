# A공정 알고리즘 설계 (v2 - 최종)

본 문서는 A공정의 Task 할당 및 레시피 선택을 위한 알고리즘들을 정의합니다. 각 알고리즘은 `src/environment/schedulers_a.py`에 구현된 클래스에 해당합니다.

---

## 1. `FIFOScheduler` (Baseline)

가장 간단한 스케줄러로, 다른 모든 지능형 스케줄러의 성능을 비교하기 위한 **성능 최저선(Baseline)** 역할을 합니다.

### 1.1. Task 선택 정책

- **선택 순서:** `rework_pool` → `wait_pool`
- **선택 방식:** 각 Pool 내에서 가장 먼저 들어온 순서대로 (First-In, First-Out).
- **로직:** 재작업이 필요한 Task를 항상 먼저 처리하여, 공정 내에 머무는 시간을 최소화하려 시도합니다.

### 1.2. Recipe 선택 정책

- **방식:** **고정 레시피 (Fixed Recipe)**
- **로직:** 장비나 부자재의 상태, 혹은 Task의 특성을 전혀 고려하지 않고, 항상 사전에 정의된 `default_recipe_A` (예: `[10.0, 2.0, 1.0]`)를 사용합니다.

### 1.3. 기대 동작 및 한계

- **기대 동작:** 시스템이 안정적으로 돌아가는지 확인할 수 있는 최소한의 기능을 제공합니다.
- **한계:**
    - 부자재나 장비가 노후화되어도 동일한 레시피를 사용하므로, 시간이 지날수록 QA 실패율(재작업률)이 급격히 증가합니다.
    - 시스템 상태 변화에 전혀 대응하지 못하여 비효율적입니다.

## 2. `AdaptiveScheduler` (Rule-Based)

`FIFOScheduler`의 한계를 개선하기 위해, **부자재의 상태(`u`)**에 따라 레시피를 동적으로 변경하는 간단한 규칙 기반 스케줄러입니다.

### 2.1. Task 선택 정책

- `FIFOScheduler`와 동일. 재작업 Task를 FIFO로 우선 처리합니다.

### 2.2. Recipe 선택 정책

- **방식:** **상태 기반 동적 레시피 (State-based Dynamic Recipe)**
- **로직:**
    1.  현재 작업을 할당할 장비의 부자재 사용량 `u` 값을 확인합니다.
    2.  `u` 값을 사전에 정의된 임계값(`u_fresh`, `u_medium`)과 비교하여, 부자재의 상태를 'fresh', 'medium', 'old' 세 가지 중 하나로 분류합니다.
    3.  분류된 상태에 매핑된 레시피를 `recipe_library`에서 선택하여 사용합니다.
        -   'fresh' 상태: 약한 레시피
        -   'old' 상태: 강한 레시피

### 2.3. 기대 동작 및 한계

- **기대 동작:** 부자재가 낡아감에 따라 레시피를 강하게 조정하여, `FIFOScheduler`에 비해 재작업률을 낮추고 안정적인 품질을 유지할 것으로 기대됩니다.
- **한계:**
    - `Task`의 개별 특성(e.g., `spec_a`가 유난히 높거나 낮은 경우)을 고려하지 않습니다.
    - 장비의 노후화(`m_age`)를 고려하지 않습니다.
    - 레시피 변경 규칙이 사전 정의된 임계값에 의존하므로 최적이 아닐 수 있습니다.

## 3. `RLBasedScheduler` (Advanced)

향후 강화학습(RL) 에이전트가 구현될 것을 대비한 인터페이스입니다. RL 에이전트는 규칙 기반 모델의 한계를 넘어, 전체 시스템의 장기적인 보상을 최대화하는 최적의 정책 `π(action | state)`을 학습하는 것을 목표로 합니다.

### 3.1. 상태 공간 (State Space)

에이전트가 관찰할 수 있는 정보의 벡터입니다. 다음을 포함할 수 있습니다.
- **대기열 정보:** `len(wait_pool)`, `len(rework_pool)`, 가장 오래된 Task의 대기 시간 등.
- **장비 상태:** 각 장비의 `status`('idle'/'busy'), `m_age`, `u`.
- **시간 정보:** `current_time`, 다음 Job 생성까지 남은 시간 (`30 - t % 30`).

### 3.2. 행동 공간 (Action Space)

에이전트가 취할 수 있는 행동의 집합입니다.
- **이산적(Discrete) 행동:**
    - **Task 선택:** 'rework 큐의 첫 번째 Task' vs 'wait 큐의 첫 번째 Task' vs 'wait 큐에서 가장 납기가 임박한 Task' 등, 사전에 정의된 Task 선택 규칙 중 하나를 고릅니다.
    - **Recipe 선택:** '보수적 레시피', '기본 레시피', '공격적 레시피' 등, 사전에 정의된 몇 가지 레시피 중 하나를 고릅니다.
- **연속적(Continuous) 행동:**
    - 레시피 파라미터 `s1, s2, s3` 값을 특정 범위 내에서 직접 결정합니다.

### 3.3. 보상 함수 (Reward Function)

행동의 좋고 나쁨을 판단하는 기준으로, 다음과 같은 요소들을 조합하여 설계할 수 있습니다.
- `+` 최종 공정(C)을 통과하여 완료된 Task 수
- `-` A공정에서 발생한 재작업(Rework) 수 (패널티)
- `-` 납기를 지키지 못한 Task 수 (Tardiness 패널티)
- `-` 부자재 교체에 들어가는 비용 (Cost 패널티)
